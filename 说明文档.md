# 说明文档

## 项目概述

`OpenAI Token Manager` 是一个偏个人运维场景的多账号 Token 管理工具。

它的核心目标不是只保存 `access_token` 和 `refresh_token`，而是把本地账号池、批量刷新、手动授权、2FA 批量授权、CPA 远端管理、Sub2API 远端管理、格式互转、导出整理、日志留痕这些环节放到同一个程序里。

当前版本已经有两条可用授权链。

第一条是协议链。它直接走协议请求，速度快，适合批量跑。

第二条是浏览器链。它会启动真实浏览器窗口，走真实页面环境，再接现有的回调换 token 逻辑，稳定性更高一些。

主程序支持源码运行，也支持打包后的 `dist/OpenAI-Token-Manager.exe` 直接运行。

## 数据目录和文件

这个项目最重要的真实数据目录是 `tokens`。

`tokens` 里保存的是本地账号池真实记录。手动授权成功、自动授权成功、2FA 批量授权成功、从 CPA/Sub2API 导入到本地、刷新成功后的最终结果，都会落到这里。

`outputs` 是派生目录，不是主数据源。这里主要放格式化导出文件、授权报告、监听结果和各种批量运行结果。常见子目录有这些。

`outputs/CPA` 保存 CPA 格式单账号导出文件。

`outputs/Sub2API` 保存 Sub2API 格式导出结果。现在整理后会生成一个带时间戳的聚合文件，文件名形如 `sub2api_accounts_20260418_171338.json`。

`outputs/auth_probe` 保存手动 OAuth 测试会话和换 token 结果。

`outputs/auth_2fa_live` 保存协议链 2FA 批量授权的单账号报告和批量汇总。

`outputs/auth_2fa_browser` 保存浏览器链 2FA 批量授权的单账号报告、批量汇总和浏览器 profile 目录。

`outputs/FlowProbeStudio` 保存 Flow Probe Studio 抓到的链路文件和调试浏览器 profile。

`dist` 是打包后的程序目录。当前打包脚本已经做了保留规则，执行 `build.py --clean` 时不会主动删掉 `dist/tokens`、`dist/outputs` 和 `dist/token_manager_config.json` 这几个运行态目录和配置文件。

## 仓库结构

`main.py` 是主程序入口，默认启动 GUI。

`token_manager/` 是主业务代码目录。GUI、配置、刷新、导入导出、远端管理、转换逻辑都在这里。

`tools/` 是辅助脚本和工具目录。这里放了 OAuth 测试脚本、协议链 2FA 授权脚本、浏览器链 2FA 授权脚本、Flow Probe 工具和登录链监听工具。

`build.py` 是标准打包脚本。

`打包.py` 是中文入口，内部还是转到 `build.py`。

`打包.bat` 是 Windows 下一键打包脚本。

`build.spec` 会由 `build.py` 自动生成。

`ico/openai.png` 是源图标，打包时会自动生成 `.ico`。

`token_manager_config.json` 是程序配置文件，GUI 里的设置页保存后会写到这里。

`version.json` 保存程序版本和功能摘要。

## 主界面模块说明

主界面分成左侧账号池、右侧功能页、底部运行日志三块。

左侧账号列表是本地 `tokens` 的主视图。这里可以刷新列表、刷新选中、刷新当前筛选结果、同步标签、删除账号、上传到 CPA 或 Sub2API，也可以做搜索和标签筛选。

右侧详情页会展示当前选中账号的详细字段，包含邮箱、计划标签、过期时间、上传状态和原始字段摘要，也能直接复制 AT 和 RT。

右侧授权页主要放手动授权和浏览器辅助自动授权。这里更适合单账号授权测试。

右侧 `2FA授权` 页是批量账号导入和批量授权入口。当前支持两种模式。

`协议链` 走纯协议请求，速度快。

`浏览器链` 会起真实浏览器窗口，一条线程对应一个浏览器实例。

右侧转换页用于预览 CPA 格式和 Sub2API 格式，也支持从文件或剪贴板导入现成 JSON 回本地 `tokens`。

右侧 `CPA` 页是远端 CPA 管理。它分远端账号池和封禁记录两个子页，支持刷新远端列表、导入回本地、刷新选中、刷新当前筛选、删除选中、禁用和启用，还能单独看封禁记录并一键删除。

右侧 `Sub2API` 页是远端 Sub2API 管理。它也分账号池和失效记录两个子页，支持远端刷新、状态筛选、分组筛选、类型筛选、删除选中、批量更新状态、处理失效账号。

右侧设置页里放基础目录、代理、线程数、OAuth 配置、浏览器链配置、CPA 配置和 Sub2API 配置。

底部运行日志会显示实时动作，不需要开控制台。长任务的异常堆栈也会落在这里。

## 本地账号数据逻辑

本地 `tokens` 永远是主数据。

程序里所有刷新、授权、导入、标签同步，本质上都是围绕本地 `tokens` 在工作。

`outputs` 只是导出或报告目录。即使清空了 `outputs`，只要 `tokens` 还在，本地账号池还在。

清理 `tokens` 功能不是简单删文件，它会保留更优的账号文件，删除重复文件，整理目录结构。这个操作会改真实数据，执行前建议确认当前账号池已经稳定。

整理导出目录功能会把本地账号重新导出成 CPA 和 Sub2API 两种格式。

CPA 会按账号一个文件导出。

Sub2API 现在会整理成一个聚合文件，文件名带时间戳，并清理旧的分散 Sub2API 单文件。

## 授权相关

### 手动授权

手动授权适合单账号排错或者你想完全自己控制浏览器过程的场景。

流程是先生成授权 URL，然后你在浏览器里登录，最后把回调 URL 贴回来。程序会用回调里的 `code` 去换 `access_token`、`refresh_token`、`id_token`，再保存到本地 `tokens`。

### 浏览器辅助自动授权

这个入口还是单账号逻辑。

程序会先启动本地回调监听，再自动打开浏览器，等你在浏览器完成登录之后，自动接住回调并保存 token。

这个模式不会自动输入账号密码，也不会自动过 2FA。它更像是单账号半自动模式。

### 2FA 批量授权的输入格式

当前 GUI 和两个批量脚本统一使用一行一个账号的格式。

```text
邮箱----密码----2FA密匙
```

中间分隔符是两个以上连字符，也就是常见的 `----`。

空行和以 `#` 开头的注释行会自动忽略。

### 2FA 批量授权的协议链

协议链对应脚本是 `tools/auth_2fa_live.py`，GUI 里的 `2FA授权 -> 协议链` 也是调这套逻辑。

这条链会做这些事。

先生成 OAuth 起点。

再通过代理和当前环境探测出口地区。

然后走 `authorize_continue`、`password_verify`、`mfa_issue_challenge`、`mfa_verify`、`workspace_select`、`organization_select`，最后拿到 callback，再换 token。

2FA 验证码默认通过 `https://2fa.live/tok/<2FA密匙>` 获取。

协议链速度快，适合高并发和批量刷授权。

它的短板也很明确。虽然已经补了部分页面侧行为，但本质上还是协议模拟，遇到更严的风控时稳定性不如浏览器链。

### 2FA 批量授权的浏览器链

浏览器链对应脚本是 `tools/auth_2fa_browser.py`，GUI 里的 `2FA授权 -> 浏览器链` 也是这套逻辑。

它会真的启动一个 Chrome 或 Edge 窗口，不是纯后台无头。

每条线程会启动一个独立浏览器实例。

每个实例有独立 profile 目录、独立调试端口、独立 cookie 和本地存储。

浏览器链当前的设计是前半段走真实浏览器环境，后半段继续复用现有 `callback -> code -> token` 逻辑。

这条链更贴近真实网页登录过程，适合在协议链不稳、需要更完整页面环境时使用。

线程数就是并发浏览器数。你如果设成 `3`，同一时间最多会有 `3` 个浏览器实例一起跑，后面的账号排队接着来。

当前版本已经做到账号级会话隔离，但还没有做到完全不同的随机 JS 指纹。也就是说，cookie、storage、profile、代理出口可以分开，但 canvas、webgl、字体、audio、GPU 这一层仍然更接近同一台机器底子。这个点如果后面继续优化，方向应该是稳定画像，不是乱随机。

### 授权报告

不管是协议链还是浏览器链，都会为每个账号生成单独报告，也会生成一个批量汇总。

单账号失败报告里通常能看到失败步、HTTP 状态、回调是否拿到、出口 IP 起点和终点、是否发生出口漂移。

协议链报告默认在 `outputs/auth_2fa_live`。

浏览器链报告默认在 `outputs/auth_2fa_browser`。

## 代理和出口 IP

主程序里的全局代理配置会被这些场景复用。

本地刷新 token。

标签同步。

CPA 上传和远端管理。

Sub2API 上传和远端管理。

协议链 2FA 授权。

浏览器链 2FA 授权。

浏览器链不是在程序里单独发代理请求，而是把代理参数传给浏览器实例本身，所以浏览器里的页面请求也会走同一套代理配置。

程序会在授权开始时探测一次出口，在结束前再探测一次。两次不一致时，会标记成 `egress_drift`。这不一定导致失败，但说明同一条授权会话前后走的出口变了，会增加风控风险。

## 订阅标签和账号状态

本地账号的标签会优先通过接口同步。

如果接口同步失败，会降级为本地 JWT 解析。

常见标签包括 `Team`、`Plus`、`Free`、`Pro`、`Enterprise` 和 `Unknown`。

CPA 和 Sub2API 远端页里还有它们自己的状态字段，比如 `active`、`error`、`pending`、`disabled`、`unavailable`、`inactive`、`unschedulable`。

Sub2API 里的 `到期暂停` 对应 `auto_pause_on_expired`。当前导出的 Sub2API 格式默认会把它写成 `true`。

## CPA 相关

CPA 导入到本地会把 CPA 格式 JSON 转回本地统一记录结构，再写入 `tokens`。

CPA 导出会按账号生成单独文件，放在 `outputs/CPA`。

CPA 远端管理页支持这些操作。

刷新远端列表。

把远端 CPA 文件导入回本地 `tokens`。

刷新选中远端账号。

刷新当前筛选结果。

删除选中远端账号。

禁用和启用远端账号。

查看封禁记录并删除。

如果远端返回的信息里带有 `Your authentication token has been invalidated. Please try signing in again.` 这一类内容，程序会把它归到封禁记录里，便于统一清理。

## Sub2API 相关

Sub2API 导出支持两种理解。

单账号预览和上传时，程序内部还是按单条格式转换。

整理导出目录时，会把所有本地账号聚合成一个 `accounts` 数组文件，再写成 `sub2api_accounts_<时间>.json`。

Sub2API 远端管理页支持这些操作。

刷新远端列表。

按分组、状态、类型筛选。

批量刷新远端账号。

批量更新状态。

删除选中账号。

单独查看失效记录并一键删除。

程序会把包含 `token invalidated` 或 `Your authentication token has been invalidated` 的记录单独识别为失效记录。

## 转换和导入导出

本地记录可以预览成 CPA 或 Sub2API 格式。

转换页支持从剪贴板或文件把 CPA/Sub2API JSON 导回本地。

CPA 转换时会补兼容 `id_token`，用于适配某些只认 CPA 结构的系统。

Sub2API 转换时会补上 `client_id`、`expires_at`、`group_ids`、`concurrency`、`priority` 和 `auto_pause_on_expired` 等字段。

整理导出目录是把当前本地全部账号重新导出，不是只导出选中项。

## Flow Probe 和辅助工具

### Flow Probe Studio

`tools/flow_probe_app.py` 是一个独立小工具，用来监听真实浏览器里的请求链。

它不是主程序的一部分，但和授权链调试很配套。

它的核心前提是，你得先有一个开启远程调试端口的 Chrome 或 Edge。

默认调试端口是 `9222`。

如果点击开始监听时报 `127.0.0.1:9222` 连接失败，意思不是链路坏了，而是本地根本没有调试浏览器在监听这个端口。

这时要么在工具里点 `启动调试浏览器`，要么手动带 `--remote-debugging-port=9222 --remote-allow-origins=*` 去启动浏览器。

### auth_probe.py

`tools/auth_probe.py` 用于单独测试 OAuth 起点、浏览器回调和换 token。

它有 `start`、`browser`、`exchange` 三个子命令。

`start` 只生成授权链接和会话文件。

`browser` 会开本地回调监听并直接等回调。

`exchange` 用现成 callback URL 去测试换 token。

### auth_2fa_live.py

这个脚本对应协议链 2FA 授权。

适合单账号命令行调试，也适合复用里面的批量逻辑。

### auth_2fa_browser.py

这个脚本对应浏览器链 2FA 授权。

适合在不经过主 GUI 时直接单账号压测浏览器授权链。

### live_login_flow_probe.py

这个脚本是更底层的登录链监听工具，用 CDP 直接附着浏览器页签记录链路。适合还原某条网页登录流程的请求顺序和关键字段。

## 运行环境

建议 Python 3.11。

当前依赖至少包括这些。

```bash
pip install -r requirements.txt
pip install pyinstaller
```

`requirements.txt` 里当前包含 `requests`、`pillow`、`curl-cffi`、`websocket-client`。

打包依赖还需要 `PyInstaller`。

## 启动方式

源码直接运行时，用这个命令。

```bash
python main.py gui
```

打包后直接运行这个文件。

```text
dist/OpenAI-Token-Manager.exe
```

## 打包方式

先只准备图标和 `build.spec`，不真正打包。

```bash
python build.py --prepare-only
```

标准打包。

```bash
python build.py --name OpenAI-Token-Manager --entry-point main.py
```

如果你就是想要一份彻底重建的产物，可以加 `--clean`。

```bash
python build.py --clean --name OpenAI-Token-Manager --entry-point main.py
```

Windows 下也可以直接双击。

```text
打包.bat
```

当前 `build.py` 会做这些事。

自动把 `ico/openai.png` 转成 `build_assets/openai.ico`。

自动生成 `build.spec`。

自动把 Tk 运行时资源和必要 DLL 带进打包产物。

调用 PyInstaller 生成 `dist/OpenAI-Token-Manager.exe`。

如果打包目标是主程序，`--clean` 时会保留 `dist/tokens`、`dist/outputs` 和 `dist/token_manager_config.json`。

如果你要打包 Flow Probe Studio，可以用类似这样的命令。

```bash
python build.py --name Flow-Probe-Studio --entry-point tools\\flow_probe_app.py --icon-png ico\\flow_probe.png --icon-ico build_assets\\flow_probe.ico
```

## 常见问题

### `ImportError: Can't connect to HTTPS URL because the SSL module is not available`

这不是接口问题，是当前 Python 环境没有带完整 SSL 组件。常见于异常裁剪过的解释器环境。换正常的 Python 或 conda 环境即可。

### `RuntimeError: Invalid admin API key`

这是 Sub2API 远端管理接口返回的鉴权失败。要检查设置页里填的是不是管理 Token 或有效 API Key，不是普通账号密码。

### `Your authentication token has been invalidated. Please try signing in again.`

这类记录会被 CPA 和 Sub2API 的失效模块单独识别出来，方便统一删除。

### 浏览器链里看到 `egress_drift`

说明同一条授权会话开始和结束的出口 IP 变了。最常见原因是代理服务同入口轮询不同出口。这会增加风控风险。

### Flow Probe Studio 提示 `HTTPConnectionPool(host='127.0.0.1', port=9222)... connection refused`

说明工具要连的调试端口没有浏览器在监听。不是链路坏了，是本地调试浏览器没启动，或者端口号对不上。

### `Rejected an incoming WebSocket connection ... remote-allow-origins`

这是 Chrome 新版对远程调试连接的来源限制。启动调试浏览器时要带上 `--remote-allow-origins=*` 或指定允许来源。

### 浏览器链报 `Failed to fetch`

如果是旧版本程序，常见原因是跳转页 URL 处理或页面内跨域 `fetch` 失败。先确认你用的是最新打包版本，再看对应失败报告里的最后一步。

### 协议链拿不到 callback

常见原因是中间某个 `continue_url` 没跟上、被风控拦住、2FA 验证没有真正通过，或者 OAuth `state` 不一致。具体看 `outputs/auth_2fa_live` 里的单账号报告。

## 建议的使用顺序

如果你只是维护本地账号池，先把账号都沉淀到 `tokens`。

如果你只是批量补授权，优先试协议链。

如果协议链不稳、需要更贴近真实页面环境，再切浏览器链。

如果你要排查网页登录细节或新增授权方式，优先用 Flow Probe Studio 或 `live_login_flow_probe.py` 把真实链路抓出来，再决定是写协议链还是浏览器链。

如果你要维护 CPA 和 Sub2API 远端池，先刷新远端列表，再做筛选、批量刷新、删除和封禁清理，最后再按需回导到本地 `tokens`。
